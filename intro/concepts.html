Concepts

The API is dealing with the following concepts: spaces, zones, capabilities and devices.

Identification

#

All the objects are carrying an identification. The identification is composed of 2 parts.

  1. Type Discrimator
  2. Random id

Type Discriminator

#

The type discrimator has two parts: a Root Type Abbreviation (RTA) and a Leaf Type Abbreviation (LFA). The RTA represents the root type of the objects.

Example: MSP.RES~01HTNWX279A3P3EB1PPD8EV2T3

Random Id

#

The id used is in a fact a ulid . More compact than a uuid, it is also lexicographically sortable based on time. So it is helping in several cases associated with efficient storage and retrieval.

Data Model

#

Person

#

The ecosystem is able to handle multiple persons. Some with accounts and some without accounts. All have a Person representation in the system. The user profile captures several pieces of information like the first and lastname of the person and the avatar that they have selected.

Spaces

#

Spaces are the basic construct that the user is used to. They respresent the different rooms and areas of the home. The spaces are hierarchical in nature and some spaces are containing other spaces. For example, a space can represent a level of the house (upstairs) and have several rooms (game room and bedrooms). A house as a location is one of the possible root spaces and represented what the space that is considered managed. All the subspaces are considered physically contiguous. There is no functional grouping done through the Spaces (like: all the bedrooms…). The root of a space tree is called a managed space and is equivalent to a Residence, an Office, a Retail...

Some API applies to both a person or a managed space. In this case, the API will refer to an Entity Id. The api will work with both a PersonId or a ManagedSpaceId.

Zones

#

Zones are representing functional aspect of the home. The zones are not necessarily matching the spaces and could span multiple spaces. Zones could be a flat list or could be hierarchical. Zones also have a relationship with spaces as they are acting in the spaces. It could impact a single space or can impact several spaces at the same time. Zones should be linked to spaces that are hierarchically dependents as the relationship with the higher space implies the relationship with all the spaces contained in that higher space. Example: When there is a motion in the kitchen, there is motion on the main floor and in the entire home but there might not be motion in the primary suite.

Capabilities

#

Capabilities are behavior that we are getting out of devices. They are acting as a layer of abstraction and the capabilities are then inscribed to a zone. It is allowing to detach the features from the actual devices delivering that feature. For example, one capability that is really looked after is Motion Detection. Motion Detection could be achieved by a PIR device, a camera device, a radar based device. By tying Motion Capability to a device, we do not care how the motion was acquired. It is also allowing different installation strategy depending on the use case for the user. For example, they might want to use a lock that a gyroscope inside to know if the door is closed or not and have the ability to lock and unlock the door. Those are two different capabilities (Lockable and Closeable). But it is possible to have two different devices instead of just one like a lock (Lockable) and a contact sensor (Closeable). Both configurations would work without any extra step from the person installing the devices.

Devices

#

Devices are way to package different set of capabilities that the user wants to add to their homes. They are the starting point in many cases but should be considered only as carrier of the features (ie. capabilities) and should never be presented to the end user unless there is a very specific need like the battery level, firmware updates or network connectivity.
